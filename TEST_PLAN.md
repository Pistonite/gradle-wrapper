# Test plan — fixture tests across Gradle versions

## Context

`gradlew` is implemented and passing 16 unit tests, but every end-to-end run so far has used **one** Gradle version (9.6.1) on **one** JDK (system Java 21). That leaves the version-dependent code paths untested — most importantly the `gradle-launcher-*.jar` fallback for pre-9.0 distributions, which has never once executed, because 9.6.1 ships both launcher jars and the code prefers `cli-main`.

Several of the projects this tool is for use Gradle releases up to ten years old, so the tool must work from **Gradle 2.0 through 9.x**. That range can't run on a single JDK: old Gradle won't run on new Java, and new Gradle won't run on old Java. Per `GRADLE_COMPATIBILITY.md`, Java 8 covers Gradle 2.0–8.14.x, which means the whole 2.x–8.x range needs exactly one JDK, with two more for 9.x.

The goal is a fixture suite that proves, for each Gradle major version, that the tool replaces an untrusted jar with a verified one and successfully runs that version of Gradle.

## Two findings that require implementation changes first

These were discovered while planning and **must be fixed before the fixtures can pass**. Both are in `src/main.rs`, in `generate_in()`.

### 1. The published wrapper checksum is not always available

`generate_in()` fetches `gradle-<v>-wrapper.jar.sha256` and aborts if it fails. But Gradle publishes that file only sporadically for older releases:

| Version | `-wrapper.jar.sha256` |
|---|---|
| 2.0, 2.1, 2.7, 2.14.1 | present |
| 3.0, 3.1, 3.2.1 | present |
| **3.3, 3.4.1, 3.5, 3.5.1** | **missing** |
| **4.0** | **missing** |
| 4.4.1, 4.10.3, 5.6.4, 6.9.4, 7.6.4, 8.14.3, 9.x | present |

So the tool is currently unusable for Gradle 3.3–4.0. Change it to **degrade gracefully**: when the checksum isn't published, accept the jar on the strength of the distribution zip we already verified, and log a warning. The trust chain still holds — verified zip in, jar generated by that verified code out. Treat only a *404* as "not published"; any other failure (network error, non-404 status) must still abort, or a transient outage silently downgrades security.

### 2. `--gradle-version` likely breaks old Gradle

`generate_in()` passes `wrapper --gradle-version <v>`, but that CLI option was added to the `wrapper` task around Gradle 4.8. Older versions require configuring the task in `build.gradle` instead, and will likely fail on an unknown option.

The flag is **redundant anyway**: we run the distribution *of that exact version*, and `gradle wrapper` with no arguments generates a wrapper pointing at the running Gradle. Dropping it should both simplify the code and fix old versions. The fixtures will confirm this — if some version still needs it, branch on version rather than restoring it globally.

## Package layout

Fixtures go in a **separate workspace member**, not `cargo test`, so `cargo test` stays fast and offline while the heavy suite is run deliberately.

```
Cargo.toml            # becomes a workspace root + the gradlew package
e2e-test/
  Cargo.toml          # package "e2e-test"
  src/main.rs         # the harness (a binary, not #[test])
.jabba/               # JABBA_HOME — project-local JDKs
.gradle-wrapper/      # GRADLE_WRAPPER_HOME under test
.gradle-test-home/    # GRADLE_USER_HOME for the inner wrapper runs
```

Run with `cargo run -p e2e-test`, optionally filtered: `cargo run -p e2e-test -- 2.14.1 9.0.0`.

A binary rather than `#[test]` functions because the suite needs ordered, readable progress output over a multi-minute run, controlled concurrency around JDK installs, and a summary table at the end — all of which fight the test harness. It exits non-zero if any fixture fails.

Add `.jabba/`, `.gradle-wrapper/`, and `.gradle-test-home/` to `.gitignore` (currently only `/target`).

## JDK provisioning

**`JABBA_HOME` must be set to `<project root>/.jabba` for every `jabba` invocation.** Without it jabba installs system-wide instead of project-local. This is the single most important detail in the harness — set it once in the process env at startup, not per-call, so no call site can forget.

The harness resolves a JDK by looking for `.jabba/jdk/<version>`; if absent it runs `jabba install <version>` and looks again. It never assumes a JDK is present. Installs are serialised behind a mutex so two fixtures needing the same JDK don't race.

**Use the `openjdk@` vendor throughout.** jabba's unqualified versions only reach down to 17, but `openjdk@` covers 8 through 26 — so a single vendor spans the whole matrix rather than mixing an unqualified 21/25 with a vendor-qualified 8. Prefer `openjdk@` over other vendors generally.

Because these version strings contain `@`, confirm what directory name jabba actually produces under `.jabba/jdk/` and match on the real directory rather than assuming the directory name equals the version string.

`JAVA_HOME` is then set to the resolved JDK directory when invoking `gradlew`, which exercises the tool's own `find_java()` path rather than depending on whatever Java is on `PATH`.

## The matrix

Three JDKs, all from the `openjdk@` vendor, cover Gradle 2.0 → 9.x. Java 8 does the heavy lifting.

| Gradle | JDK | Why this version |
|---|---|---|
| 2.0 | `openjdk@8` | the actual floor |
| 2.14.1 | `openjdk@8` | last 2.x |
| 3.5.1 | `openjdk@8` | last 3.x — **and has no published wrapper checksum**, so it tests the fallback |
| 4.10.3 | `openjdk@8` | last 4.x |
| 5.6.4 | `openjdk@8` | last 5.x |
| 6.9.4 | `openjdk@8` | last 6.x |
| 7.6.4 | `openjdk@8` | last 7.x |
| 8.14.3 | `openjdk@8` | last 8.x — Java 8's upper bound |
| 9.0.0 | `openjdk@21` | first 9.x; `gradle-gradle-cli-main` layout |
| 9.6.1 | `openjdk@25` | current release on the latest LTS |

Every version from 2.0 through 8.14.3 uses `gradle-launcher-*.jar`, so this finally exercises the fallback branch in `find_launcher_jar()`.

## What each fixture asserts

For version *V* on JDK *J*, in a fresh project directory under `target/e2e/<V>/`:

1. Write `gradle/wrapper/gradle-wrapper.properties` with a `distributionUrl` for *V*, and plant a **deliberately invalid** `gradle-wrapper.jar` containing plain text.
2. Run `gradlew --version --no-daemon` with `JAVA_HOME=J`, `GRADLE_WRAPPER_HOME`, and `GRADLE_USER_HOME` all pointed at the project-local directories.
3. Assert:
   - exit status is success
   - stdout contains `Gradle <V>` — proves the correct version actually ran, not just that something ran
   - the project's `gradle-wrapper.jar` is no longer the planted text
   - where Gradle publishes one, the jar's SHA-256 **equals the published `gradle-<V>-wrapper.jar.sha256`**; where it doesn't, assert the run still succeeded and a warning was emitted
   - `.gradle-wrapper/known-good/gradle-wrapper-<V>.jar` now exists
   - the per-run `work/` directory was cleaned up
4. Run a **second time** and assert idempotence: no download, no copy, and the project files are byte-identical to after the first run.

## Behaviour tests

Run once against a single cached version (9.6.1), independent of the matrix:

- **Tamper** — overwrite the jar with garbage; it is silently restored to known-good.
- **`-all` downgrade** — an `-all.zip` `distributionUrl` still resolves, and the project's properties are replaced with Gradle's own `-bin` file.
- **Nonexistent version** (99.99.99) — non-zero exit, nothing written to `known-good/`, work directory preserved for inspection.
- **Exit-code passthrough** — a failing task propagates non-zero.
- **`JAVA_HOME` invalid, and unset with no `java` on `PATH`** — both reproduce the shell script's exact messages and exit 1.
- **Empty `GRADLE_WRAPPER_HOME=""`** — currently returns `Some("")`, so cache directories are created relative to the CWD, polluting the project. `find_java()` already treats empty `JAVA_HOME` as unset; `wrapper_home()` should match. Test the fixed behaviour.
- **Never writes `gradlew`/`gradlew.bat`** — the `wrapper` task emits those into the stub project, and copying them into the project would overwrite the binary with a shell script. Assert the project's `gradlew` is unchanged after a run.

## Cost and caching

A cold run downloads 3 JDKs (~600MB) plus 10 distributions, each fetched **twice** — once by us for generation, once by the generated wrapper jar into `GRADLE_USER_HOME`. Roughly 2–3GB and several minutes.

All three caches live at the project root and persist between runs, so only the first run pays. Old distributions are much smaller than modern ones (2.x is ~30MB vs 9.x at 141MB), so the real total is well under the worst case. Pass `--no-daemon` on every invocation so a ten-version sweep doesn't leave ten daemons running.

## Verification

1. `cargo test` — the existing 16 unit tests still pass, still offline and fast.
2. `cargo run -p e2e-test -- 9.6.1` — single known-good version, fastest signal the harness works.
3. `cargo run -p e2e-test -- 2.0` — the riskiest case: oldest Gradle, oldest JDK, `gradle-launcher` fallback, and the `--gradle-version` removal all at once.
4. `cargo run -p e2e-test -- 3.5.1` — the missing-wrapper-checksum fallback path.
5. `cargo run -p e2e-test` — full matrix; expect a summary table and exit 0.
6. Re-run the full matrix to confirm every fixture takes the cached path and stays idempotent.
7. `git status` clean afterwards — confirms the new `.gitignore` entries cover all three cache directories.

Record findings in `LOG.md` as the work proceeds, per the existing convention.
